## 第一章：为什么需要「外挂式」AI（0:00 - 1:30）

### 开场 Hook

（黑屏，渐显白色文字，打字机效果）

> 一个 SaaS 应用平均有一百多个功能界面。用户真正会用的，不超过十五个。

（切画面：一个布满菜单项和按钮的复杂后台界面，鼠标在屏幕上犹豫地来回移动）

### 痛点 1：页面即迷宫

你花了一年时间做的系统，功能齐全。但用户登录之后，面对满屏的菜单、表格、筛选条件、操作按钮——他不知道从哪里开始。

"批量导入在哪？""上个月的数据怎么筛？""这个灰色的按钮能不能点？"

如果你做的是 ERP、WMS 这类制造业系统，情况更糟——系统部署到客户工厂，产线上的工人根本不看操作手册。他们在客户群里一遍遍问同样的问题。你不得不扩招客服。系统越来越重，支撑团队也跟着膨胀。

用户不是不想用。他是真的找不到。

最后他关掉了页面。你失去了一个用户，而你开发的那些高级功能——他从来不知道它们存在。

### 痛点 2：客服在回答同样的问题

（切画面：客服后台，满屏的工单列表，滚动展示）

"怎么改收货地址？""订单怎么取消？""报表在哪儿导出？"

这些问题，帮助文档里写得清清楚楚。但没人看文档。他们选择问。你的客服团队每天 60% 的精力消耗在重复问答上。业务在增长，客服成本线性和用户数同步爬升——这不是一个可规模化的模式。

### 痛点 3：AI 落地太难

你知道 AI 能解决这个问题。你研究了市面上的方案：

- **对话机器人**：只能聊天，不能操作页面。用户说"帮我导出报表"，它只会回复"请点击左上角菜单 → 报表管理 → 导出"。用户听完还是不会。
- **浏览器插件**：让客户装插件？IT 部门第一个跳出来反对。而且插件看不到你系统的业务逻辑，只能做最浅层的自动化。
- **自建方案**：LangChain、RAG、向量数据库、Agent、Prompt Engineering……一个完整的团队，两个月起步。你算了一下 ROI，默默关掉了浏览器标签页。

你缺的不是技术能力。你缺的是一个**能直接嵌入任何网站、既能回答问题、又能操作页面的外挂式 AI**。

### 一句话定义

（画面：干净的产品 Logo 居中，副标题逐字浮现）

> VoltaireAI —— 一行 `<script>`，为任何网站注入 AI 智能助手。

（切画面：一个普通的企业后台页面。右下角浮现精致的蓝色浮动按钮。鼠标移上去显示"AI 助手"。点击弹出 420×520 的对话框——干净、现代、不喧宾夺主）

### 它不只是聊天机器人

（分屏对比，左侧灰色调，右侧蓝色调）

| 传统聊天机器人 | VoltaireAI |
|---|---|
| 回答预设 FAQ，问什么都绕回那几句 | 理解页面结构，基于你的知识库灵活回答 |
| 不能触碰页面，只能"告诉你"怎么做 | 直接执行 JS，点击按钮、填写表单、跳转页面 |
| 需要 SDK 集成，适配各种前端框架 | 一行 `<script>` 标签，不改任何现有代码 |
| 答案靠人工逐条维护，越用越僵 | DOM 自动扫描 + 文档持续上传，知识库自增长 |

### 四大核心能力

（四个圆形图标依次弹出，每个停留 2 秒，伴随音效）

1. **智能问答** — 用户问什么，AI 从你的知识库中找到答案，不是模板回复
2. **意图识别** — 自动判断用户想"问问题"还是"做操作"，走不同的处理管线
3. **代码生成与执行** — 操作请求直接生成 JavaScript，在用户浏览器里运行，不是截图不是录屏
4. **失败自修复** — 代码跑出错了？错误自动回传 AI，重新生成修正代码，用户无需任何操作

## 第三章：全景演示 — 它到底怎么用（2:45 - 4:45）

### 场景设定

（画面：一个典型的后台管理系统——用户列表、搜索框、批量操作按钮、分页组件）

用户登录系统后，右下角浮动按钮自动出现。我们来看几个真实的对话场景。

### 场景 A：智能问答

用户打开对话框，输入："这个页面是做什么的？"

后端发生了什么（画面一角的画中画展示数据流）：

```
用户消息
 ↓
意图分类（LLM） → "question"
 ↓
Agent 选择 → qa_agent
 ↓
知识库查询（同时检索 sitemap / table / document 三类库）
 ↓
LLM 精选最相关元素
 ↓
生成回答
```

不到两秒，AI 回复：

> 这是用户管理页面。你可以通过左侧搜索框按姓名或邮箱筛选用户，右上角的蓝色按钮用于批量导出数据。表格每行的操作列支持编辑和删除。需要我帮你执行具体操作吗？

用户追问："帮我导出最近注册的用户。"

### 场景 B：分步操作执行

AI 判断意图为"operation"，切换处理管线。然后它不是回复一段文字——而是给出了一个**结构化的操作计划**：

> **第一步**：在日期筛选区域设置时间范围。这一步的目的是限定导出的数据范围，避免导出全部历史数据。
>
> `[一个可执行的代码块，包含具体操作]`
>
> 旁边有一个蓝色的 **「执行」** 按钮。

用户点击执行 → 页面上的日期输入框自动填入 `2026-01-01` → 按钮变为绿色 ✓

> **第二步**：点击"筛选"按钮，加载符合时间条件的用户。这一步确保表格中只显示目标数据。

用户点击执行 → 页面自动点击筛选按钮 → 表格刷新，只显示今年的用户

> **第三步**：点击右上角"批量导出"，选择导出格式。这一步完成最终的数据导出。

用户点击执行 → 浏览器弹出下载对话框

三步完成。每一步都有**文字说明**（告诉用户为什么做这一步）和**独立可执行的代码**。用户可以逐个点击执行，也可以一次性执行所有步骤。用户在操作中学会了操作。

### 场景 C：失败自动修复

（制造一个边界情况：页面按钮实际文字是"批量匯出"，AI 第一次用的是"批量导出"）

代码执行失败。控制台报错，提示找不到目标元素。

前端自动将错误信息发回后端。AI 收到错误上下文后，不是简单地重试同一段代码——它重新分析：

> 上一次执行失败，找不到"批量导出"按钮。检查知识库中的页面元素，发现该按钮实际文字为"批量匯出"。重新生成代码，改用模糊匹配策略……

新代码使用 `screen.getByText((content) => content.includes('匯出'))`。再次执行——成功。

用户全程只做了一件事：**点了执行按钮**。重试、修正、重新执行——AI 在背后自动完成。

## 第四章：技术特色深度解析（4:45 - 6:30）

### 特色 1：完全配置驱动的 Agent 系统

（画面：管理后台的 Agent 编辑界面，切换到代码视图展示 agents.json）

AI 的所有行为——如何分类意图、如何生成代码、用什么风格回复——全部由 `agents.json` 配置文件控制。没有一行规则是硬编码在 Python 代码里的。

```json
{
  "id": "operation_agent",
  "role": "你是网页自动化操作导师……",
  "rules": [
    "将复杂操作拆分为多个独立步骤",
    "每个步骤用 ```js-start ... ```js-end 包裹",
    "禁止使用原生 DOM 选择器，必须使用 Testing Library API",
    "每个代码块末尾必须执行一个用户可见的操作"
  ]
}
```

管理后台直接编辑 → 保存 → 自动热加载。下一次对话立即生效，无需重启服务。这意味着你可以针对不同业务场景定制不同的 Agent——ERP 系统用一套提示词，电商后台用另一套——精确控制 AI 的输出行为，比微调模型更灵活、更透明。

### 特色 2：三类知识库协同工作

（画面：三个圆环交汇的架构图，每个圆环内显示示例数据）

知识库不是一个大杂烩。它按用途分成三类，各司其职：

| 知识库 | 存什么 | 怎么来 | 角色 |
|---|---|---|---|
| **站点地图** (sitemap) | 页面上的按钮、链接、输入框，含 CSS 选择器、坐标、可用操作 | DOM 自动扫描 / Playwright 探索器 | 让 AI 知道"页面上有什么" |
| **操作指引** (table) | 业务流程的步骤序列，如"创建采购订单的 5 个步骤" | 文档上传 / 手动录入 | 让 AI 知道"业务流程怎么做" |
| **文档资料** (document) | FAQ、产品说明、帮助文档 | 文件上传（支持 11 种优化策略） | 让 AI 知道"怎么回答知识性问题" |

**核心机制：每次对话，三类知识库同时查询。** LLM 从候选结果中精选最相关的内容，注入生成上下文。AI 始终知道自己"站在哪个页面上，能操作什么元素，业务怎么流转"。

ChromaDB 向量数据库不可用时（比如 Windows 上 sqlite3 版本不兼容），自动回退到 JSON 文件 + LLM 关键词分解搜索——保证功能在任何环境下都能正常运行。

### 特色 3：11 种全自动 RAG 优化策略

（画面：策略卡片依次排列，高亮推荐项）

文档上传不是简单的"切成片段塞进库"。系统内置 11 种 RAG 优化策略，全程 LLM 自动处理，无需人工干预：

- **层次化索引**（推荐 | 评分 0.84）：先让 AI 生成 300 字内容摘要，再拆细粒度 Chunk。检索时先匹配摘要层，再定位细节层——精准度远超直接分块
- **融合检索**（评分 0.82）：语义搜索和关键词匹配双路并行，结果融合排序——兼顾语义理解和精确命中
- **知识图谱 RAG**（评分 0.78）：从文档中提取实体和关系，构建结构化的知识图谱。不是检索文本，而是检索"谁和什么相关"
- **CRAG**（评分 0.82）：检索后评估每个 Chunk 的相关性，低分的自动标记并补充外部信息——保证返回给你的永远是最相关内容

每种策略由 **LLM 子 Agent 并行处理**。大文档自动拆分为 3000 字批次，4 个子 Agent 同时运行。你只负责上传文件，AI 完成剩下的一切。

### 特色 4：智能站点探索器

（画面：Playwright 浏览器窗口，自动点击导航菜单，页面切换，新元素被高亮扫描）

日常使用中，站点的 DOM 元素由 voltaire.js 自动扫描上传。但如果你想一次性摸清整个网站的交互地图，内置的 Playwright 站点探索器能完成这件事——**而且它不只扫描首页**。

工作流程：

> 浏览器打开目标网站 → 扫描当前页面所有可交互元素 → LLM 分析哪些按钮和链接值得点击（自动排除注销、删除、提交等危险操作） → 点击 → 新页面加载 → 重新扫描新出现的元素 → 继续探索下一层……

它能处理 SPA 动态内容——点击导航标签，等待异步加载完成，扫描新出现的区域。支持三种登录认证模式：自动填写表单 / 加载已保存的 Cookie / 手动登录轮询等待。

一个 50 个页面的网站，15 分钟完成全站交互元素扫描，生成完整站点地图 JSON，自动上传到知识库。你甚至不需要打开浏览器——全程命令行或 API 驱动。

## 第五章：安装与集成（6:30 - 7:15）

### 运行环境

- **必需**：Python 3.10+、任意兼容 OpenAI API 的 LLM 密钥（DeepSeek / OpenAI / Ollama / vLLM 均可）
- **可选**：Node.js（仅本地打包 `@testing-library/dom` 时需要）、Chrome/Chromium（仅使用站点地图工具时需要）

### 三步部署

**第一步：启动后端**

```bash
git clone https://github.com/Rootchmod/VoltaireAI.git
cd VoltaireAI
pip install -r requirements.txt
```

编辑 `backend/config/settings.json`，填入 API 密钥：

```json
{
  "model_sources": [{
    "name": "deepseek",
    "base_url": "https://api.deepseek.com",
    "api_key": "sk-你的密钥",
    "model": "deepseek-chat"
  }],
  "default_model_source": "deepseek"
}
```

```bash
cd backend && python main.py
# 服务运行在 http://localhost:8001
```

**第二步：在你的网站中加一行**

在目标页面 `</body>` 前插入：

```html
<script src="http://localhost:8001/voltaire.js"></script>
```

刷新页面。右下角浮动按钮出现。完成。

**第三步（可选但推荐）：完善知识库**

打开 `frontend/admin.html`，通过可视化后台：

- 上传帮助文档和 FAQ（选择合适的优化策略）
- 运行站点地图生成器，自动扫描全站交互元素
- 调整 Agent 提示词，定制 AI 的行为风格

### 生产部署

```bash
# 多 worker 模式
uvicorn main:app --host 0.0.0.0 --port 8001 --workers 4

# voltaire.js 部署到 CDN 或 Nginx 静态目录
# CORS 白名单限制允许的域名
```
