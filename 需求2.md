知识库功能要做一下修改

1. 知识库要做分类，有表格式，文档式，和站点地图式
2. 知识库不需要做目标站点标记，因为这个项目的目标本来就计划做在单网站上
3. 站点地图主要用来存储站点上的按钮、输入框等可操作句柄信息
4. 表格式主要用来做流程化操作指引，比如用户不会操作入库，需要多步操作，就需要用户上传操作指引到表格式文档里
5. 文档式主要存储一些操作文档，用户提问时如果匹配到教程可以回复，另外在解析文档式知识库时，如果有给ai用的操作指引，就存一份对应的操作指引到一个默认的表格式库里
6. 表格式、文档式、站点地图都要有一个默认库，库的内容可以被编辑或删除单条，但是默认库不能全库删除，类似系统级知识库。
7. 知识库要永久存储，不能每次重启都重新导入，用户重启项目或者刷新页面，都要把之前上传的知识库带入进来
8. 在知识库上传界面，添加知识库优化方案的下拉框，下拉框展示策略名称和评分，多种优化策略见【附录1】，用户选择好一个方案后要在旁边标注好核心思路
9. 在上传和构建知识库时要按照对应的优化策略进行处理，默认使用层次优化索引。知识库优化处理可以拉起多个子agent来处理，不要用规则处理

【总体原则】
- 所有操作必须通过 原生 JavaScript 完成
- 不使用任何测试框架（QUnit / Mocha / Jest / Testing Library）
- 每一步代码必须 独立、可执行、在浏览器中产生可见效果

【代码结构规则】
- 将复杂操作拆分为多个独立步骤
- 每个步骤前用自然语言说明：做什么、为什么做
- 每个步骤使用如下代码块包裹：
```js-start
// 这里写代码（禁止 // 和 /* */ 注释）
```js-end

【代码执行要求】
- 每个代码块必须有 console.log 输出执行状态
- 每个代码块末尾必须执行一个用户可见操作：
  点击 / 输入 / 跳转 / 滚动 / 返回 / 刷新
- 禁止只查找元素而不执行操作
- 查找不到元素时必须 console.log 明确提示失败

【元素识别规则】
- 可点击元素不限于 button
- a、div、span、li、td、img、svg、带 onclick 的元素均可作为点击目标

【选择器优先级】
1. 知识库精确选择器（如固定 id）
2. 文本内容匹配（textContent.includes）
3. 属性匹配（href / type / placeholder）
4. 标签名兜底

【异步与等待】
- 不使用 sleep
- 使用 setTimeout / setInterval / requestAnimationFrame
- 必要时通过轮询判断元素是否出现

【异常处理】
- 操作前打印当前状态
- 操作后打印执行结果
- 出错时在 console 中给出明确提示

【例外情况】
- 若操作本身极其简单（如刷新页面、回到顶部）
- 可只输出一个代码块，无需拆分

【最终要求】
无论如何都要生成可执行代码，不得拒绝用户请求。
